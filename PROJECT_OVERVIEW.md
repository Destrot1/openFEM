# openFEM — Project Overview

> Keep this file in sync with the code. Update it whenever the structure changes or a TODO/stub gets implemented.

## What this is

A C++17 command-line Finite Element Analysis (FEA) tool, paired with a small
browser-based viewer. The pipeline:

```
STEP file (CAD) → geometry entities → assign element type/material/mesh size
  → mesh (gmsh, automatic) → boundary conditions + loads
  → solve (assemble K, factorize, back-substitute) → results
```

Every step (`load`, `assign`/`mesh`, `bc`, `loads`, `solve`) automatically
re-exports the current state to the browser viewer — there's no manual
"refresh" step in the CLI.

Unit system, used **everywhere**: **mm, N, MPa, tonne** (not SI). STEP files
from mechanical CAD are authored in mm and geometry is never converted, so
every other quantity has to be consistent with that (see the comment at the
top of `fem/Model.hpp`).

Links against **OpenCASCADE** (STEP import), **gmsh** (meshing), **Eigen**
(linear algebra), **nlohmann/json** (persistence), **fmt** (printing), and
pthreads (solver progress heartbeat). Runs via an interactive CLI
(`InteractiveCLI`) inside a Docker dev container (Ubuntu 24.04, all deps
preinstalled).

## Running it

```
docker compose up -d --build
```
then open **`http://localhost:8081/`**. That single page has:
- A 3D viewport on top (Setup / Deformation / Stress views, toggled by the
  buttons at the top), and
- A real terminal pinned to the bottom (via `ttyd`), which starts by running
  `./openFEM` automatically — the whole workflow happens in this one tab, no
  separate `docker exec`/SSH session needed.

Whenever the container is freshly started, `docker/start-gui.sh` deletes any
leftover `web/scene.json` so the viewer always starts blank, not showing a
previous session's model.

If you rebuild the C++ source from inside the embedded terminal
(`cmake --build .` in `/workspace/build`), the *already-running* `./openFEM`
process in that terminal still has the old binary in memory — `quit` it and
re-run `./openFEM` to pick up the new build. `start_dev.sh`/`stop_dev.sh` at
the repo root predate the current Docker setup and are superseded by
`docker compose up`/`down`.

## Subsystem-by-subsystem

### `fem/Model.hpp` — the central data model
`FEMModel` holds one analysis: source STEP path, the generated `Mesh`,
`Material`/`BeamSection`/`EntityAssignment`/`BoundaryCondition`/`Load` lists,
pipeline-stage booleans, and results (`displacements`, `nodalDisplacements`,
`nodalStressNormal`, `nodalVonMises`, `maxVonMisesRaw`, `yieldStrengthUsed`,
`safetyFactor`).

- `Material.yieldStrength` (MPa, 0 = not set) drives the safety-factor
  calculation; round-trips through `save`/`open`.
- `EntityAssignment` binds a geometry entity (`dim`+`entityId`, gmsh's own
  tag) to an element type, material, thickness/section, and mesh size.
- `BoundaryCondition.locked[6]` is a lock/free mask (ux,uy,uz,rotx,roty,rotz)
  — `FIXED`/`PINNED` are presets for it, `CUSTOM` lets you pick per-DOF.
  No prescribed nonzero displacement, only locked-or-free.
- `Load` is `FORCE_NODAL` (Fx/Fy/Fz in N) or `PRESSURE` (MPa). `GRAVITY`
  exists in `LoadType` but is intentionally not offered in the CLI — it's a
  body force, doesn't fit the per-entity flow.

### `geometry/StepImporter` — STEP import (OpenCASCADE)
`load()` reads a STEP file via `STEPControl_Reader`. `exportToBRep()` writes
it to BREP (lossless, topological — no read-order ambiguity like STEP), which
is what `Mesher` actually imports into gmsh.

⚠ Known issue (flagged in code): the loaded `TopoDS_Shape` is a file-scope
global, so the importer is a de-facto singleton — one geometry at a time.

### `mesh/Mesher` — mesh generation (gmsh)
Two phases:
- **`importGeometry(brepFile)`**: imports the BREP via
  `gmsh::model::occ::importShapes` + `synchronize()`, discovers entities
  (solids/free faces/free edges, by gmsh's own `(dim, tag)` — no separate
  numbering anywhere in this project), with a bounding box per entity for
  display. `allBoundaryEntities()` returns the *unfiltered* list (including a
  solid's own faces/edges) for `bc`/`loads`, which target a specific
  face/edge, not a whole solid.
- **`meshAndExtract(assignments)`**: sets the smallest requested
  `meshSize` as a global `Mesh.MeshSizeMax` ceiling, generates with a 20s
  timeout in a child process (`fork`/`waitpid`/`kill` — gmsh has no safe
  in-process interrupt) and falls back to gmsh's own unconstrained sizing if
  it doesn't finish in time, then reads nodes/elements back per-assignment
  (each element correctly tagged with *its own* entity's material/thickness/
  section, not always `assignments[0]`).
- `exportWebView(dir, bcs, loads, ResultsExport)` writes `web/scene.json`:
  nodes, per-entity triangles/lines, bc/load markers, and (if present)
  displacements/stress/safety-factor. `getBoundingBox()` calls are wrapped in
  try/catch — can throw "Empty bounding box" for a stray entity left behind
  after `gmsh::open()` reloads a sized `.msh` result.
- A quick, unconstrained preview mesh (sized off the model's own bbox
  diagonal, clamped to [0.5, 20]mm) is auto-generated the first time
  `exportWebView` runs with no real mesh yet, so `load` shows *something*
  immediately, before `assign`/`mesh` ever run.
- Shell support (`FEMHint::SHELL`) is **disabled in the CLI**: the old
  "thin solid → suggest shell" heuristic is removed because, without real
  mid-surface extraction, choosing shell on a solid did nothing geometrically
  (gmsh still meshed the full volume into tetrahedra). The underlying
  `ShellElement`/`ElementType::TRI3`/`Assembler` TRI3 path is untouched and
  ready for when mid-surface extraction is implemented — see `cmdAssign()`.

### `elements/` — per-element stiffness matrices
Common interface `Element` (`stiffnessMatrix`, `numNodes`, `dofPerNode`):
- **`TetraElement`** (Tet4, 4×3 DOF): constant-strain `B`/`D`,
  `Ke = Bᵗ D B · volume`. Also exposes `stress(nodes, E, nu, ue)` = `D·B·ue`,
  used by `StressResolver` (TETRA4 only — shell/beam stress isn't computed
  anywhere in this project).
- **`ShellElement`** (Tri3, 3×6 DOF): CST membrane + simple bending, 18×18
  `Ke`. ⚠ Not reachable from the CLI right now (see above); ⚠ no drilling
  stiffness for `rotz` (zero diagonal block — singularity risk if ever
  assembled).
- **`BeamElement`** (Beam2, 2×6 DOF): 12×12 Euler-Bernoulli local stiffness,
  rotated to global. ⚠ No defined roll angle (arbitrary y/z orientation about
  the beam axis); ⚠ section properties (area/Iy/Iz/J) are entered by hand in
  `assign`, not computed from geometry.

### `fem/Assembler` — global stiffness matrix
`computeDOFMap()` gives each node 3 or 6 DOF (max over touching elements) and
a cumulative offset table (`nodeOffset()`/`nodeDOF()`, used by `LoadResolver`/
`StressResolver`/the CLI to translate node-level data to/from global DOFs).
`assemble()` builds each element's local `Ke` and scatters it via triplets
into a sparse `K`.

### `fem/LoadResolver` — entity-scoped BC/Load → global DOF
`resolveFixedDOFs()`: for each `BoundaryCondition`, looks up its entity's
nodes (`Mesher::entityNodeIndices`) and locks `locked[d]` DOFs.
`resolveForces()`: for `FORCE_NODAL`, splits the total force across the
entity's nodes by **tributary area/length** (triangle/line connectivity from
`Mesher::surfaceTriangles`/`surfaceLines`), not equally — denser mesh regions
get proportionally less force per node, matching how a real distributed load
would integrate. For `PRESSURE`, the area-weighted average face normal gives
the direction. ⚠ The pressure normal's *sign* depends on gmsh's triangle
winding, not verified to always point outward — check the viewer before
trusting it.

### `fem/StressResolver` — post-processing
`computeElementStresses()`: per TETRA4 element, `sigma = D·B·ue` (6
components) and von Mises, from the solved displacement vector.
`averageToNodes()`: averages every element's stress onto its nodes for a
smooth contour (`nodalSigmaNormal`/`nodalVonMises`), **and separately** keeps
the true unaveraged peak von Mises + the safety factor
(`yieldStrength / maxVonMisesRaw`, using the yield strength of whichever
element produced that peak) — averaging would otherwise understate a sharp
stress concentration (e.g. at a hole's edge).

### `fem/Solver` — linear solve
Penalty method for BCs (`max(diag(K))·1e12` added to each fixed DOF, force
zeroed), then `Eigen::SparseLU` (direct solver, ~100k DOF practical ceiling).
Factorization runs on a background thread while the main thread prints an
elapsed-time heartbeat every 5s (no way to get real incremental progress out
of Eigen's SparseLU otherwise). ⚠ Only homogeneous (zero) prescribed
displacement is supported, not arbitrary values.

### `io/ModelSerializer` — JSON persistence of the model *setup*
`save`/`open` round-trip `name`, `stepFile`, `materials` (including
`yieldStrength`), `sections`, `assignments`, `bcs`, `loads`. This is the
model **definition** only — not the mesh or any solved results (those come
back by re-running `assign`/`solve`). For saving/restoring a *solved* state
to view later without re-solving, see `snapshot`/`restore` below.

### `cli/InteractiveCLI` — the user-facing loop
A REPL (`run()`) that prints pipeline status (with a hint of which command
to run next for any unset step) and dispatches commands:

| Command | Behavior |
|---|---|
| `load` | STEP path → `StepImporter` → BREP → `Mesher::importGeometry` → lists entities. Auto-exports to the viewer (shows an immediate quick preview mesh). |
| `assign` | Per entity: element type (**tetra/beam only** — shell hidden, see above), mesh size, material (`C40`/`aluminium`/`custom`, with yield strength), beam section (hand-entered). Then **automatically meshes** (`cmdMesh()` is no longer a separate required step) and auto-exports. |
| `mesh` | Re-mesh with the current assignments — only needed to redo it (e.g. after a sized attempt timed out and fell back to gmsh's default sizing). |
| `bc` | Per (dim,tag) entity (unfiltered list, with center coords shown): `fixed`/`pinned`/`custom` lock mask. |
| `loads` | Per entity: `force` (Fx/Fy/Fz, N) or `pressure` (MPa). |
| `solve` | `Assembler` + `LoadResolver` + `Solver`, fills displacements/stress/safety factor, auto-exports. Prints peak von Mises + safety factor. |
| `results` | ❌ stub — the web viewer's Deformation/Stress pages cover this visually; no text summary yet. |
| `save` / `open` | Model **setup** JSON round-trip (see ModelSerializer above). |
| `snapshot` | Copies the current `scene.json` (mesh+results) to `web/snapshots/<name>.json`. |
| `restore` | Lists saved snapshots, copies one back to `scene.json` — the browser picks it up on its own (polling), no re-solving. |
| `status` / `help` / `quit` | |

`view`/`web` commands from earlier in development are **removed** — every
state-changing command re-exports on its own (`autoExportWeb()`), and the
browser polls `scene.json`'s `Last-Modified` header (~1.2s) to pick up
changes without a manual refresh.

### `tests/test_beam_frame.cpp`
Hand-built 3-beam planar frame (no STEP/mesh/CLI), exercises
`Assembler`/`Solver` directly. Confirms the element/assembly/solve math
end-to-end in isolation. ⚠ No analytical reference value/assertion — printed
output, eyeballed.

## Web viewer (`web/index.html`)

Single page, Three.js, no build step (loaded straight from `index.html` via
an import map). Layout: a 3D viewport on top, a fixed-height real terminal
(`ttyd`, see "Running it") pinned to the bottom.

Three modes, toggled by buttons in the viewport:
- **Setup**: hover a face/edge to see its `(dim, tag)` name (raycasting);
  entities with a constraint/load tinted purple/yellow; bc/load markers +
  legend (forces show every nonzero Fx/Fy/Fz component, not just the
  dominant axis); a small XYZ **gizmo** in the bottom-right corner mirrors
  the camera's rotation only (not zoom/pan) — like a CAD nav-cube — instead
  of a model-anchored `AxesHelper`.
- **Deformation**: exaggerated deformed shape (scaled so max displacement ≈
  15% of model size), 4 quadrants (Ux/Uy/Uz/Total), one shared camera. A
  single gizmo (with X/Y/Z labels) sits centered exactly on the crosshair
  where the 4 quadrants meet.
- **Stress**: same deformed shape, 4 quadrants (σx/σy/σz/Von Mises), same
  crosshair gizmo (no labels, to limit clutter). The Von Mises legend also
  shows the true unaveraged peak stress and the safety factor as text.

Both result modes use the same discretized 9-band colormap (blue → cyan →
light cyan → 3 greens → yellow → orange → red), not a smooth gradient — hard
band edges read more like a contour plot. `scene.json` is polled (HEAD +
`Last-Modified`), and the whole scene is torn down and rebuilt on every
detected change (so a second `mesh`/`solve` replaces the old view instead of
piling on top of it) — no manual browser refresh needed, ever, *except* after
editing `index.html` itself (only `scene.json` is polled).

## What works end-to-end today

`load` → `assign` (auto-meshes) → `bc` → `loads` → `solve`, all auto-exporting
to the browser viewer at every step, for a single-material TETRA4 solid mesh.
Verified on a perforated plate (C40 steel, 50kN tension): converges, reports
a plausible peak von Mises and safety factor against the assigned yield
strength.

## Known gaps

In rough order of how often they'd actually bite:
1. **No real shell support** — disabled in the CLI; needs mid-surface
   extraction (OpenCASCADE) to mean anything geometrically. Revisit before
   re-enabling it as a user-facing choice.
2. **Pressure load direction unverified** — sign may need flipping depending
   on gmsh's triangle winding for a given model.
3. **Beam section properties** hand-entered, not computed from geometry
   (same underlying gap as #1 — needs 2D cross-section extraction).
4. **Stress only for TETRA4** — shell/beam elements (once shell is
   reinstated) won't show anything on the Stress page.
5. **Homogeneous BCs only** — no prescribed nonzero displacement.
6. **Shell drilling DOF** is zero stiffness (singularity risk if ever
   assembled — irrelevant while shell is disabled).
7. `cmdResults()` text stub — viewer covers the visual side.
8. `StepImporter` is a singleton — one geometry loaded at a time.
